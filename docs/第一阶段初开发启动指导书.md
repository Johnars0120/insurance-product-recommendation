# 保险产品智能推荐系统第一阶段初开发启动指导书

## 一、阶段定位

第一阶段是项目从“文档规划”进入“代码开发”的启动阶段。本阶段不追求一次性完成全部功能，也不追求界面和模型效果一步到位，核心目标是让全组成员在同一套仓库、同一套目录结构、同一套协作规则下开始工作，并尽快跑通一个最小可运行版本。

第一阶段的关键词是：统一环境、明确分工、跑通骨架、形成闭环。

本阶段结束时，项目应该达到以下状态：

1. 所有组员都能访问 GitHub 私有仓库。
2. 所有组员都能使用 GitHub Desktop 克隆项目并切换到自己的功能分支。
3. 本地 FastAPI 项目可以正常启动。
4. 数据与算法、后端接口、前端页面、文档测试四个方向都有明确负责人。
5. 系统至少具备首页或健康检查接口，后续模块可以在此基础上继续开发。
6. 第一阶段任务、接口约定、每日同步方式已经写入项目文档。

## 三、第一阶段目标

第一阶段的目标不是完成完整推荐系统，而是完成“开发启动”和“最小可协作基础”。

具体目标包括：

1. 建立统一仓库  
   项目代码、原始数据、设计文档、开发说明、测试记录都统一放入 GitHub 仓库中管理，避免通过压缩包反复传文件造成版本混乱。

2. 建立统一开发环境  
   所有组员使用相同的项目目录结构、依赖文件 `requirements.txt` 和启动命令，保证每个人本地运行结果基本一致。

3. 建立统一分支规则  
   `main` 分支只保存稳定代码，组员分别在自己的功能分支上开发，功能完成后通过 Pull Request 合并。

4. 建立最小 FastAPI 骨架  
   项目至少包含 `app/main.py`、`/health` 接口、基础测试文件和后续模块目录，为第二阶段功能开发提供入口。

5. 建立协作文档  
   将分工、协作规则、接口说明、任务看板、启动会议要求写入 `docs/` 目录，方便组员随时查看。

## 四、仓库目录说明

当前项目采用以下目录结构：

```text
insurance-product-recommendation/
├─ app/
│  ├─ main.py
│  ├─ config.py
│  ├─ database.py
│  ├─ routers/
│  ├─ services/
│  ├─ models/
│  ├─ templates/
│  └─ static/
├─ data/
│  ├─ raw/
│  ├─ processed/
│  └─ output/
├─ saved_models/
├─ docs/
├─ tests/
├─ requirements.txt
├─ README.md
└─ .gitignore
```

各目录作用如下：

- `app/main.py`：FastAPI 应用入口。
- `app/config.py`：项目路径、数据路径、目标字段等配置。
- `app/database.py`：后续 SQLite 数据库连接配置。
- `app/routers/`：页面路由和 API 路由。
- `app/services/`：数据处理、模型训练、评估、推荐等业务逻辑。
- `app/models/`：数据库模型或数据结构定义。
- `app/templates/`：Jinja2 页面模板。
- `app/static/`：CSS、JS、图片等静态资源。
- `data/raw/`：原始数据文件，包括 `data.xlsx` 和 `eval.xlsx`。
- `data/processed/`：处理后的中间数据，不作为主要提交内容。
- `data/output/`：推荐结果导出文件，不作为主要提交内容。
- `saved_models/`：训练后的模型文件，通常不提交大模型文件。
- `docs/`：项目文档、协作说明、测试记录和答辩资料。
- `tests/`：自动化测试。

## 五、角色分工

第一阶段建议按照四个方向分工。如果组员人数较少，可以一人兼任两个方向。

### 1. 项目协调负责人

主要职责：

- 管理 GitHub 仓库和协作者权限。
- 确认每位组员所在分支和负责模块。
- 组织启动会议和每日进度同步。
- 审核 Pull Request，避免不稳定代码直接进入 `main`。
- 协调接口、字段、目录结构等公共约定。

第一阶段交付：

- 组员名单和分工表。
- GitHub 协作者添加完成。
- 每日进度同步机制建立。
- 第一阶段任务看板更新。

### 2. 数据与算法负责人

主要职责：

- 读取 `data/raw/data.xlsx` 和 `data/raw/eval.xlsx`。
- 确认样本数量、字段数量、目标字段和正负样本比例。
- 设计数据预处理方案。
- 确认第一批模型：逻辑回归、决策树、随机森林。
- 确认评估指标：Accuracy、Precision、Recall、F1、AUC。
- 为后端提供统一训练和预测函数。

第一阶段交付：

- 数据概览说明。
- 目标字段说明。
- 模型训练初步方案。
- 需要后端展示的数据字段清单。

### 3. 后端接口负责人

主要职责：

- 熟悉 FastAPI 项目结构。
- 确认 `app/main.py` 可以运行。
- 规划 `/data`、`/train`、`/evaluate`、`/recommend` 页面路由。
- 规划数据概览、模型训练、模型评估、推荐预测等 API。
- 和算法负责人约定服务函数输入输出。

第一阶段交付：

- 后端路由规划。
- API 路径和返回字段初稿。
- 能正常启动的 FastAPI 项目。
- 与算法模块的对接需求说明。

### 4. 前端展示负责人

主要职责：

- 规划系统首页、数据管理页、模型训练页、评估页、推荐页。
- 确认页面需要展示哪些数据和图表。
- 使用 Jinja2 模板和静态资源实现第一版页面结构。
- 为后续 ECharts 图表预留展示区域。

第一阶段交付：

- 页面结构草稿。
- 每个页面需要的数据字段说明。
- 首页或基础模板初稿。
- 图表展示计划。

### 5. 文档测试负责人

主要职责：

- 维护 `docs/` 下的协作说明、接口说明和任务看板。
- 记录启动会议内容。
- 设计测试记录模板。
- 保存运行截图和问题记录。
- 为后续答辩材料积累过程证据。

第一阶段交付：

- 成员分工记录。
- 启动会议记录。
- 测试记录模板。
- 第一阶段开发日志。

1. 统一项目目标。

## 七、GitHub Desktop 协作流程

### 1. 接受仓库邀请

组员先把 GitHub 用户名发给项目负责人。负责人添加协作者权限后，组员登录 GitHub 接受邀请。

仓库地址：

```text
https://github.com/Johnars0120/insurance-product-recommendation
```

能打开仓库内容，说明权限已经生效。

### 2. 克隆项目

使用 GitHub Desktop：

```text
File -> Clone repository -> URL
```

填写仓库地址：

```text
https://github.com/Johnars0120/insurance-product-recommendation
```

本地路径建议选择：

```text
F:\工程实践II\insurance-product-recommendation
```

### 3. 切换分支

根据分工切换到自己的分支：

- `feature/data-model`：数据与算法
- `feature/backend`：后端接口
- `feature/frontend`：前端展示
- `feature/docs`：文档测试

组员不要直接在 `main` 分支上开发。

### 4. 同步代码

每次开始开发前，在 GitHub Desktop 中点击：

```text
Fetch origin
```

如果出现 `Pull origin`，继续点击同步。

### 5. 提交修改

每完成一个小功能，在 GitHub Desktop 左侧查看修改文件，在 Summary 中填写提交信息，例如：

```text
feat: 完成数据概览统计功能
fix: 修复 Excel 读取路径问题
docs: 更新接口说明
test: 添加模型评估测试记录
```

然后点击：

```text
Commit to feature/xxx
```

### 6. 推送分支

提交后点击：

```text
Push origin
```

### 7. 创建 Pull Request

功能完成后创建 Pull Request：

```text
base: main
compare: 自己的 feature 分支
```

PR 描述中说明：

- 本次完成了什么。
- 修改了哪些主要文件。
- 如何运行或测试。
- 是否需要其他同学配合。

## 十二、常见问题处理

### 1. 组员打不开仓库

先确认是否已经接受 GitHub 邀请。如果还不能访问，请检查组员发来的是否是正确 GitHub 用户名。

### 2. 不知道改哪个分支

根据分工选择：

- 数据与算法：`feature/data-model`
- 后端接口：`feature/backend`
- 前端展示：`feature/frontend`
- 文档测试：`feature/docs`

### 3. 不会命令行

优先使用 GitHub Desktop 完成克隆、切换分支、提交和推送。运行项目时仍需要使用 PowerShell 或 VS Code 终端。

### 4. 依赖安装失败

先确认已经激活虚拟环境：

```powershell
.venv\Scripts\activate
```

再重新安装依赖：

```powershell
pip install -r requirements.txt
```

### 5. 端口 8000 被占用

可以临时换成 8001：

```powershell
uvicorn app.main:app --reload --host 127.0.0.1 --port 8001
```

### 6. main 分支出现不稳定代码

不要继续在 `main` 上修来修去。应先切换到对应功能分支修复，测试通过后再通过 Pull Request 合并。

## 十三、第一阶段结束后的下一步

第一阶段完成后，项目进入第二阶段：核心功能开发。

第二阶段建议开发顺序：

1. 数据概览功能  
   展示样本数、字段数、目标字段、正负样本比例。

2. 模型训练功能  
   支持训练逻辑回归、决策树、随机森林等基线模型。

3. 模型评估功能  
   展示 Accuracy、Precision、Recall、F1、AUC。

4. 推荐预测功能  
   输出客户推荐概率和推荐等级。

5. 页面可视化  
   使用 ECharts 展示样本分布、指标对比和推荐等级分布。

第一阶段最重要的成果不是“功能多”，而是让所有人进入同一个开发节奏：仓库一致、分支一致、目标一致、交付一致。只要这个基础打牢，后续功能开发会顺畅很多。
