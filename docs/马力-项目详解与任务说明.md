> 本文档面向马力，详细解释项目每个部分的位置和作用，说明你该做什么，以及答辩环节的准备思路。

## 一、项目全景（这是什么项目？）

这是一个**保险产品智能推荐系统**，大学课程设计项目。目标是做一个轻量级 Web 系统，跑通从"读取数据 → 训练模型 → 评估模型 → 推荐预测 → 页面展示 → 导出结果"的完整流程。

简单一句话：**输入客户数据，输出"该客户是否可能购买移动房车险"的推荐结果。**

### 技术路线一览

| 层级 | 技术 | 作用 |
| --- | --- | --- |
| 后端框架 | FastAPI | 提供 Web 服务和 API 接口 |
| 前端模板 | Jinja2 | 动态渲染 HTML 页面 |
| 前端图表 | ECharts | 柱状图、饼图等可视化 |
| 数据库 | SQLite | 存储训练记录、评估指标、推荐结果 |
| 数据处理 | Pandas / openpyxl | 读写 Excel 数据 |
| 机器学习 | scikit-learn | 训练逻辑回归、决策树、随机森林 |
| 模型持久化 | joblib | 保存/加载训练好的模型 |
| 测试 | pytest + httpx | 接口测试 |

### 2.2 `app/` — 应用核心代码

这是整个系统的核心代码目录，采用分层架构：

| 文件 | 作用 |
| --- | --- |
| `app\main.py` | **系统入口文件**。创建 FastAPI 应用实例，目前只有 `/` 首页和 `/health` 健康检查两个端点。后续所有路由都会注册到这里。启动命令：`uvicorn app.main:app --reload` |
| `app\config.py` | **全局配置文件**。定义所有路径常量：`DATA_DIR`（数据目录）、`SAVED_MODEL_DIR`（模型保存目录）、`TRAIN_DATA_FILE`（训练数据路径）、`EVAL_DATA_FILE`（评估数据路径）、`TARGET_COLUMN`（目标字段："移动房车险数量"）。 |
| `app\database.py` | **数据库配置**。定义 SQLite 数据库文件路径和连接 URL（`insurance_recommendation.db`）。 |
| `app\routers\` | **路由模块**（当前为空，待开发）。负责定义 URL 路由，接收 HTTP 请求并返回响应。例如：`/data` 数据页面路由、`/api/models/train` 训练 API 路由。 |
| `app\services\` | **业务逻辑模块**（当前为空，待开发）。负责数据处理、模型训练、预测等核心业务逻辑，供 routers 调用。例如：读取 Excel、训练模型、计算评估指标。 |
| `app\models\` | **数据模型模块**（当前为空，待开发）。定义数据库表结构（SQLAlchemy ORM 模型），例如：训练记录表、评估指标表、推荐结果表。 |
| `app\templates\` | **前端模板目录**（当前为空）。存放 Jinja2 HTML 模板文件。后端渲染数据后生成页面返回给浏览器。 |
| `app\static\` | **静态资源目录**（当前为空）。存放 CSS、JS、图片等静态文件。ECharts 的 JS 脚本也可以放这里。 |

### 2.3 `data/` — 数据目录

## 三、协作分支架构

项目在 GitHub 上使用 4 个功能分支协作：

| 分支 | 负责人方向 | 主要任务 |
| --- | --- | --- |
| `feature/data-model` | 数据与算法 | 数据读取、预处理、训练模型（逻辑回归/决策树/随机森林）、评估指标、保存最优模型、封装预测函数 |
| `feature/backend` | 后端接口 | 编写 FastAPI 路由和 API、设计 SQLite 表、整合 service 层 |
| `feature/frontend` | 前端展示 | Jinja2 页面模板、CSS 样式、ECharts 图表、推荐结果展示 |
| `feature/docs` | 文档测试（**你的分支**） | 记录分工、编写测试、保存截图、整理答辩材料 |

> 组长在 `main` 分支上做整体协调和代码合并。

## 五、答辩环节说明

### 5.1 答辩中你需要讲什么

作为文档测试负责人，答辩时你的重点：

1. **你做了什么**：测试验证、截图记录、文档整理、答辩材料准备。
2. **协作方式**：团队如何使用 GitHub 功能分支协作（每人一个分支，PR 合并），你如何通过测试确保系统质量。
3. **项目流程**：你是整个流程的"见证者"和"记录者"，可以最完整地描述整个项目的开发过程。

### 5.2 答辩演示流程建议

| 环节 | 时间 | 内容 |
| --- | --- | --- |
| 开场 | 1 分钟 | 项目背景：保险产品销售需要精准推荐，本系统用机器学习预测客户购买概率 |
| 架构 | 1 分钟 | 展示目录结构图、技术栈（FastAPI + scikit-learn + ECharts） |
| 演示 | 3-4 分钟 | 现场运行系统，依次展示：数据概览 → 模型训练 → 评估指标对比 → 推荐结果 |
| 分工 | 1 分钟 | 展示团队分工表，说明协作流程（GitHub 分支 + PR） |
| 总结 | 1 分钟 | 项目收获、遇到的问题和解决方案 |

### 5.3 答辩常见问题准备

| 问题 | 回答方向 |
| --- | --- |
| 为什么用 FastAPI？ | 轻量、自动生成 Swagger 文档、异步支持、适合小型 Web 系统 |
| 为什么正负样本比例约 6%:94% 很重要？ | 类别不平衡，只看 Accuracy 没意义（全预测负样本也有 94% 准确率），必须看 Precision/Recall/F1/AUC |
| 团队如何协作？ | GitHub Flow：每人功能分支开发、PR 合并、每日同步进度 |
| 你的核心贡献是什么？ | 编写测试用例、验证系统功能、整理文档、准备答辩材料、确保交付质量 |

## 七、总结

马力，你的角色是**文档测试负责人**。简单说就是：

- **记录者**：把团队分工、开发过程、接口文档都整理清楚
- **验证者**：按验收标准测试系统，确保每个功能都能跑通
- **截图者**：保存所有页面的运行截图作为交付物
- **答辩者**：准备答辩 PPT 和演示流程，在答辩时完整展示项目

这个项目的 MVP（最小可用版本）要求是：系统能启动 → 能读数据 → 能训练模型 → 能显示评估 → 能生成推荐。你的任务就是逐条验证这些功能，并记录好一切，最终在答辩中呈现出来。
