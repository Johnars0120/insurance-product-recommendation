# 算法实验补充说明

本文档用于补充 `保险产品推荐.pdf` 中的数据分析、类别不平衡处理、模型对比和实验结论部分。当前 Web 系统已经完成数据概览、模型训练、模型评估、推荐生成、历史记录和导出功能；本文档重点说明系统背后的算法实验依据，便于写入设计说明书或答辩讲解。

## 1. 实验任务

本项目面向保险客户推荐场景，目标是根据客户属性和历史投保信息，预测客户是否可能购买移动房车险。原始目标字段为 `移动房车险数量`，建模时将其转换为二分类标签：

- `移动房车险数量 > 0`：记为正样本，表示客户购买过移动房车险。
- `移动房车险数量 = 0`：记为负样本，表示客户未购买移动房车险。

因此，本项目本质上是一个二分类推荐问题。模型输出客户购买概率后，系统再按照概率区间生成高意向、中意向、低意向推荐等级，辅助业务人员安排跟进顺序。

## 2. 数据集概况

当前项目使用的训练集和评估集均来自课程数据文件，字段结构一致。

| 数据集 | 样本数 | 字段数 | 正样本 | 负样本 | 正样本比例 | 缺失值 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 训练集 `data.xlsx` | 5822 | 86 | 348 | 5474 | 5.98% | 0 |
| 评估集 `eval.xlsx` | 4000 | 86 | 238 | 3762 | 5.95% | 0 |

从数据分布可以看出，购买移动房车险的客户比例不足 6%，正负样本差异明显。这说明该任务存在类别不平衡问题。如果只使用 Accuracy 作为评价标准，模型可能通过大量预测为负样本取得较高准确率，但并不能真正找出潜在购买客户。

因此，系统在模型评估中同时展示 Accuracy、Precision、Recall、F1 和 AUC，重点关注模型对少数类客户的识别能力。

## 3. 数据探索结论

训练集共有 86 个字段，字段唯一值数量最少为 2，最多为 40，平均唯一值数量约为 7.33，说明数据中包含大量离散型或分箱后的客户画像特征。

从数值特征偏度看，部分历史投保字段分布高度偏斜，例如：

- `投保冲浪险`
- `投保冲浪险数量`
- `投保卡车险数量`
- `投保农机险数量`
- `投保身残险数量`

这些字段多数属于低频保险产品或低频历史购买行为，只有少量客户存在对应记录。因此，模型不能简单假设所有特征都服从均匀或正态分布，需要使用适合表格分类任务的模型进行综合判断。

从与目标字段的相关性看，和移动房车险购买行为相关性较高的字段主要包括：

- `投保车险`
- `投保车险数量`
- `投保船险数量`
- `个人第三方保险`
- `购买力水平`
- `投保火险`
- `平均收入`
- `第三方私人险数量`

这些字段说明，客户已有的车辆相关保险、财产类保险、第三方责任险和收入购买力信息，对移动房车险推荐具有一定参考价值。也就是说，客户是否购买移动房车险并不是随机事件，而是与其风险偏好、资产类型、购买能力和历史投保行为有关。

## 4. 数据预处理方式

当前系统采用 sklearn Pipeline 组织预处理和模型训练，保证训练阶段和预测阶段使用完全一致的数据处理逻辑。

主要处理方式如下：

- 数值字段：使用中位数填充缺失值，再进行标准化处理。
- 类别字段：使用众数填充缺失值，再进行 One-Hot 编码。
- 目标字段：将 `移动房车险数量` 转换为是否购买的二分类标签。
- 训练和预测：通过同一个 Pipeline 串联预处理器和分类模型，避免手工处理训练集和评估集时出现字段不一致问题。

课程 PPT 中提到可以通过相关系数筛选特征并保存预处理 CSV。当前系统采用工程化 Pipeline 方式直接完成预处理，优点是流程稳定、可复用，并且能在 Web 推荐接口中直接使用。如果需要完全复现 PPT 的离线实验形式，可以额外导出 `train_preprocess.csv` 和 `test_preprocess.csv` 作为补充材料，但这不是当前系统运行所必需的步骤。

## 5. 类别不平衡处理

由于正样本比例约为 6%，本项目在模型训练中使用类别权重平衡策略：

```python
class_weight="balanced"
```

该策略会在训练过程中提高少数类样本的权重，使模型更加关注潜在购买客户。相比直接上采样或下采样，类别权重方式不改变原始样本数量，适合在 Web 系统中作为稳定、轻量的处理方案。

答辩时可以这样说明：

> PPT 中展示了上采样和下采样思路，本系统采用 sklearn 模型内置的类别权重平衡机制处理类别不平衡问题。由于购买移动房车险的客户占比不足 6%，系统不只展示 Accuracy，而是同时展示 Precision、Recall、F1 和 AUC，以便更全面地评价推荐模型。

## 6. 模型选择

当前系统实现了三类代表模型：

- 逻辑回归：作为线性基线模型，训练速度快，结果稳定。
- 决策树：作为可解释性模型，便于理解客户分层规则。
- 随机森林：作为集成模型，通常比单棵决策树更稳定，适合提升整体预测效果。

PPT 中还出现了 MLP、KNN、GaussianNB、AdaBoost 等模型。当前系统没有全部接入这些算法，原因是本项目重点是构建完整推荐系统闭环，而不是堆叠所有算法。三类模型已经覆盖线性模型、树模型和集成模型，能够支撑模型对比和业务推荐演示。

需要注意的是，系统中的决策树使用 sklearn 的 `DecisionTreeClassifier`，不是手写 ID3。答辩时建议表述为：

> 系统实现了 sklearn 决策树作为可解释性基线模型。PPT 中的 ID3 属于理论和实验扩展，本系统工程实现采用 sklearn 决策树，以保证稳定性、可维护性和与其他模型统一的训练接口。

## 7. 当前模型对比结果

使用当前样例数据训练并在评估集上测试，三类模型指标如下：

| 模型 | Accuracy | Precision | Recall | F1 | AUC |
| --- | ---: | ---: | ---: | ---: | ---: |
| 逻辑回归 | 0.6935 | 0.1122 | 0.6008 | 0.1892 | 0.7184 |
| 决策树 | 0.5737 | 0.0950 | 0.7227 | 0.1679 | 0.6987 |
| 随机森林 | 0.8535 | 0.1754 | 0.3950 | 0.2429 | 0.7251 |

从结果可以看出：

- 随机森林的 Accuracy 和 AUC 较高，整体区分能力较好。
- 决策树的 Recall 较高，能够覆盖更多潜在购买客户，但 Precision 较低。
- 逻辑回归表现较稳定，适合作为基线模型。
- 各模型的 Precision 普遍不高，主要原因是正样本比例过低，推荐任务本身存在明显类别不平衡。

因此，模型选择不能只看 Accuracy。对于保险推荐业务，如果目标是尽可能覆盖潜在客户，可以更重视 Recall；如果目标是减少无效触达，可以更重视 Precision；如果希望综合比较模型区分能力，可以重点参考 AUC 和 F1。

## 8. 参数设置说明

当前系统使用较稳定的参数配置：

- 逻辑回归：`max_iter=1000`，保证模型有足够迭代次数完成收敛。
- 决策树：`max_depth=5`，限制树深度，降低过拟合风险。
- 随机森林：`n_estimators=100`、`max_depth=8`，兼顾训练速度和模型稳定性。

这些参数不是为了追求单次实验指标最高，而是为了让系统在演示、测试和推荐生成时表现稳定。若继续扩展实验，可以增加不同 `max_depth`、不同正则化方式、不同随机森林树数量的对比记录。

## 9. 系统落地说明

与 PPT 中偏 Notebook 的算法流程相比，当前项目进一步实现了 Web 化业务闭环：

- `/data` 页面展示数据概览和样本分布。
- `/train` 页面支持选择模型并启动训练。
- `/evaluate` 页面展示五类指标、训练历史和模型对比图。
- `/recommend` 页面生成客户推荐名单、推荐概率、推荐等级和推荐理由。
- `/docs` 页面提供 Swagger API 文档，便于接口测试和系统集成。
- SQLite 保存训练记录、指标记录和推荐结果。
- CSV 导出功能支持后续人工复核和汇报展示。

因此，项目不仅完成了 PPT 中的保险购买预测任务，还把算法结果封装成了可操作、可展示、可复用的推荐系统。
